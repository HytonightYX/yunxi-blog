React新特性(一) Context,lazy,Suspense和memo

React更新到16.8之后，出现了大量的新特性，本系列文章是我作为一名React初学者通过网上资料学习新特性的笔记，而且加入了一些自己的思考，难免会有疏漏。如果发现有错误，恳请告知，不胜感激！

目前打算分为两个部分，第一部分即这篇文章，第二部分为hooks的学习笔记

Context 上下文

  • 提供了一种能让数据在组件树中传递而不必通过props一层层手动向下传递
  • 类似store？
  • 滥用会造成组件高耦合

例子如下：

class Leaf extends Component {
    render() {
        return (
            <BatteryContext.Consumer>
                {battery => (
                    <OnlineContext.Consumer>
                        {
                            online => <h1>Battery: {battery}, Online: {String(online)}</h1>
                        }
                    </OnlineContext.Consumer>
                )}
            </BatteryContext.Consumer>
        )
    }
}

class Middle extends Component {

    render() {
        return (
            <Leaf/>
        )
    }
}

class App extends Component {
    state = {
        battery: 60,
        isOnline: false
    }

    render() {
        const {battery, isOnline} = this.state

        return (
            <BatteryContext.Provider value={battery}>
                <OnlineContext.Provider value={isOnline}>
                    <button
                        onClick={() => {
                            this.setState({battery: battery - 1})
                        }}
                    >
                        减1
                    </button>

                    <button
                        onClick={() => {
                            this.setState({isOnline: !isOnline})
                        }}
                    >
                        切换联网
                    </button>
                    <Middle/>
                </OnlineContext.Provider>
            </BatteryContext.Provider>
        )
    }
}

使用生产者和消费者的模型,实现数据的传递

ContextType 语法糖

直接使用Context会有一些缺点

  • 大量使用context会使组件很不纯粹
  • 使用是不是太麻烦了？比如要获取一个数据（例如电量信息battery）要先写一个Consumer标签包裹住代码，然后其中写一个函数来获取
<BatteryContext.Consumer>
                {battery => (
                    <OnlineContext.Consumer>
                        {
                            online => <h1>Battery: {battery}, Online: {String(online)}</h1>
                        }
                    </OnlineContext.Consumer>
                )}
</BatteryContext.Consumer>

于是，我们可以用ContextType在render方法之前提前获取好context

class Leaf extends Component {

 static contextType = BatteryContext

 render() {
  const battery = this.context

  return (
   <h1>Battery: {battery}</h1>
  )
 }
}

但是，这种方法只能订阅单一context。

要使用多个Context，可以参考官网，其实还是上面那种略复杂的方法

Lazy 懒加载

  • 要用时再加载
  • 途径
  • import 切分
/**
 * 懒加载
 * lazy的返回就是一个 React组件
 */
const About = lazy(() => import('./About.jsx'))

class App extends Component {
    render() {
        return (
            <About/>
        )
    }
}

但是，这样会报错

Error: A React component suspended while rendering, but no fallback UI was specified.

因为Lazy懒加载期间会有一个loading的过程，React便不知道在loading时该渲染些什么，因此，就要用到下面这个Suspense

Suspense 填充Lazy时的loading空挡

  • 内置组件，专门用来配合Lazy
  • 将lazy进来的组件包起来，并设置好fallback属性即可
const About = lazy(() => import('./About.jsx'))
const Loading = () => <h1>Loading...</h1>
class App extends Component {
    render() {
        return (
            <Suspense fallback={<Loading/>}>
                <About/>
            </Suspense>
        )
    }
}

刷新后，可以很明显的看到先出现“Loading…”一闪而过，然后再渲染出About组件了。

打开Chorme开发者工具的Network面板，你看，About组件就变为一个单独的请求了。

当然，如果webpack默认命名不好辨认，你也可以加一行注释，来自定义文件名

const About = lazy(() => import(
    /* webpackChunkName: "about" */
    './About.jsx')
)

Wait，发生错误怎么办？

是的，既然是单独的网络请求，那么这个网络请求就有可能失败。那么这个组件的错误将会导致整个应用崩溃。所以我们要用到错误边界(Error Boundaries)

static getDerivedStateFormError() {
 return {
  hasError: true
 }
}

render() {
 if (this.state.hasError) {
  return <div>Error</div>
 }

 return (
  <Suspense fallback={<Loading/>}>
   <About/>
  </Suspense>
 )
}
进一步，可不可以在加载错误后自动重试？

memo 优化渲染性能

React的做了个什么事情呢，用数学公式来表达，就是$UI=f(data)$

例如：

{data: 1} => <div>1</div>

如果data变为2呢？这时便重新render，

{data: 2} ==render ==> <div>2</div>

那么，会不会有这个情况，data没改变，但是也会触发组件的重新渲染呢？如果是，那么这是不是一种性能的浪费呢？答案是肯定的。

思考一个常见的父子组件场景。App为父组件，其下有一个子组件Foo，如果App重新渲染了，Foo会不会也重新渲染呢？写个代码来模拟一下

class Foo extends Component {
 render() {
  console.log('Foo rendered')
  return 
 }
}

class App extends Component {
 state = {
  count: 0
 }

 render() {
  return (
   <div>
    {/*这个按钮一定会让App重新渲染*/}
    <button onClick={
     () => {
      this.setState({
       count: this.state.count + 1
      })
     }
    }>Re-Render App</button>
    <Foo name={'YunXi'}/>
   </div>
  )
 }
}

由于按下按钮后改变了App组件的state，因此一定会让App重新渲染

而Foo组件每次渲染都会在控制台打印一句话”Foo rendered”，运行代码后是这样的

你看，第一次打印是首次渲染，第二次打印则证明了父组件的Re-Render也会导致子组件的Re-Render，但是子组件其实没有必要渲染。下面给出两种方案

1.生命周期函数 shouldComponentUpdate

class Foo extends Component {
 shouldComponentUpdate(nextProps) {
  return nextProps.name !== this.props.name;
 }

 render() {
  console.log('Foo rendered')
  return <div>I'm Foo</div>
 }
}

意思是，只有当传入的props的name属性改变了，才触发Foo的重渲染。如果你试过，你会发现控制台确实只打印一次“Foo rendered”了

2.PureComponent

React.PureComponent 与 React.Component 很相似。两者的区别在于 React.Component并未实现 shouldComponentUpdate()，而 React.PureComponent 中以浅层对比 prop 和 state 的方式来实现了该函数。

class Foo extends PureComponent {
 render() {
  console.log('Foo rendered')
  return <div>I'm Foo</div>
 }
}

这种方式会有两个陷阱，关键就是浅层对比四个字

陷阱一：传入对象

比如传入的props是一个复杂的对象，例如一个user信息

user = {
  name: 'YunXi',
  age: 20
}

那么，如果按下按钮，会将user的年龄加1，那你需要这么做

class Foo extends PureComponent {
    render() {
        console.log('Foo rendered')
        return <div>{this.props.user.age}</div>
    }
}

class App extends Component {
    state = {
        count: 0,
        user: {
            name: 'YunXi',
            age: 20
        }
    }

    render() {
        const user = this.state.user
        return (
            <div>
                <button onClick={
                    () => {
                        const _user = JSON.parse(JSON.stringify(this.state.user))
                        user.age++
                        this.setState({user: _user})
                    }
                }>Re-Render App</button>
                <Foo user={user}/>
            </div>
        )
    }
}
陷阱二：传入function
class App extends Component {
    state = {
        count: 0,
        user: {
            name: 'YunXi',
            age: 20
        }
    }

    // 箭头函数
    callback = () => {}

    render() {
        return (
            <div>
                <button onClick={
                    () => {
                        this.setState({count: this.state.count + 1})
                    }
                }>Re-Render App</button>
                {/* 如下这样不会触发Foo的重新渲染 */}
                <Foo user={this.state.user} cb={this.callback}/>
            </div>
        )
    }
}

3.memo

首先，将Foo改写为函数组件

function Foo(props) {
 console.log('Foo rendered')
 return <div>{props.user.name}</div>
}

引入memo

import React, { Component, memo } from 'react'

用memo将Foo包裹起来，这里我做简化处理，用箭头函数直接取代掉原先Foo了

const Foo = memo(
    (props) => {
        console.log('Foo rendered')
        return <div>{props.user.name}</div>
    }
)

这样，Foo便不会做无意义的重渲染了
